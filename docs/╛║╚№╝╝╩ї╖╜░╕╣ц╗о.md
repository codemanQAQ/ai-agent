## 3. 数据工程与特征治理（评分项 3.1）

### 3.1 商品数据模型

> 50–100 条脱敏电商数据是评委给的"必用资源"，自己再爬 200 条丰富类目（服装/数码/美妆/家电/食品）。

```text
catalog_spu(id, title, brand, category_path, price, stock, attributes_json,
            description_md, images[], video_url, status, version, updated_at)
catalog_chunk(spu_id, chunk_type[TITLE|DESC|ATTR|REVIEW|IMAGE], text, token_count)
```

**关键设计：**

- **三种 chunk 共存**：标题/属性/正文/评论 4 类文本 chunk + 图像 chunk。检索阶段按 chunk_type 加权（属性 1.0 / 标题 0.9 /
  评论 0.6）。
- **属性结构化抽取**：上线时跑一次 LLM 抽取，把"防水、容量 14 寸、防泼"沉淀到 `attributes_json`，索引到 Milvus
  metadata（用于反选过滤）。
- **价格/库存一致性**（评分点）：走现有 `indexing` 的 Outbox 模式 — 商品表更新 → 写 outbox → MQ 消息 → Milvus metadata
  局部更新（不重算 embedding）。这一手能直接对应"数据一致性保障"评分点。
- **Chunking 策略**：用 `RagOpenAiTokenCounter` 现成的，按 256 token 滑窗 + 60 token 重叠；属性类不切分整体写入（提高召回精度）。

### 3.2 多模态向量化

Doubao-embedding-vision 是统一 text+image 空间，所以图像 chunk 与文本 chunk 写**同一个 Milvus collection**，多模态混排自然成立。索引时：

```
商品.images[] → Doubao-embedding-vision(image) → vector + metadata(spu_id, chunk_type="IMAGE")
商品.title    → Doubao-embedding-vision(text)  → vector + metadata(spu_id, chunk_type="TITLE")
```

> 这一点是评委容易眼前一亮的地方："拍照找货"和"以文搜图"用的是同一张表，不需要双索引切换。

## 4. Agent 编排（评分项 3.2，决战核心）

新增模块 `agent`，目录建议：

```
com.bytedance.ai.agent
├── api/         AgentTurnRequest / StreamEvent / ToolCallView
├── application/ AgentOrchestrator(主流程) / TurnRecoveryTask
├── intent/      IntentClassifier (LLM/规则混合) / IntentType 枚举
├── slot/        SlotExtractor / SlotSchema(JSON Schema) / NegationSlotExtractor
├── planner/     ToolPlanner（基于 Spring AI 的 Tool Calling）
├── tool/        ToolRegistry + 各 Tool 实现（search/compare/exclude/cart/image_search/...）
├── verify/      GroundedAnswerVerifier (引用对齐 + 幻觉拦截)
└── memory/      ConversationMemory (短期) / UserProfileMemory (长期)
```

### 4.1 主流程（单 turn 时序）

```
1. POST /agent/turn (SSE) ───── 携带 conversationId + 文本/图片
2. 短期记忆装配（最近 N=6 轮 + 用户画像摘要）
3. IntentClassifier ── 规则前置（关键词 / 价格表达式）+ LLM 兜底
   → 输出 IntentType + confidence
4. SlotExtractor (function_call schema)
   → 抽出 must / must_not / range / category / scenario / image_ref
5. Planner 选 Tool（依据 IntentType + Slot 状态机表，下文 4.2）
6. ToolExecutor 并发执行（Reactor），过程中流式回写
   - search_products / compare_products / exclude_products
   - image_search / add_to_cart / list_cart / place_order
7. Verifier:
   - 引用 SPU 必须存在于本 turn 召回集合
   - 价格/库存必须从 catalog 实时读，不从 LLM 文本取数
   - 不允许编造 SKU / 优惠券
8. AnswerGenerator 用 RAG 上下文 + Tool 结果 → 流式 token + 卡片
9. 写 outbox / 更新会话 / 落 trace
```

### 4.2 Intent → Tool 调度表

| 意图                 | 必触发工具                                     | 典型场景           | 评分维度     |
|--------------------|-------------------------------------------|----------------|----------|
| `RECOMMEND_VAGUE`  | `search_products`                         | 推荐适合油皮的洗面奶     | 基础       |
| `FILTER_BY_ATTR`   | `search_products` (with structured slot)  | 200 以下蓝牙耳机     | 基础       |
| `REFINE`           | `search_products` (in last result subset) | 要轻量的           | 进阶       |
| `COMPARE`          | `compare_products` (并行 N)                 | A 和 B 哪个保湿     | 进阶       |
| **`EXCLUDE`**      | `search_products` (must_not 注入)           | 推荐防晒霜，但不要酒精和日系 | **高级加分** |
| `SCENARIO_BUNDLE`  | `scenario_outfit` (跨类目 fan-out)           | 三亚度假从防晒到穿搭     | 高级加分     |
| **`CART_OP`**      | `cart_*` StateMachine                     | 把刚才那款加到购物车     | **高级加分** |
| **`IMAGE_SEARCH`** | `image_search` (vision embedding)         | 上传图片 + 我要这种外套  | **高级加分** |
| `OUT_OF_SCOPE`     | `decline_with_reason`                     | 帮我写代码          | —        |

> 推荐用 **Spring AI 的 Tool Calling**（已经在 pom 里）+ Doubao 的 function calling 能力。Planner 不是手写的 if-else，而是用一个
> system prompt + tool schema 让 LLM 选工具，**这一点直接对应"Prompt Engineering"评分点**。

### 4.3 Prompt Engineering（直接是评分点）

每个工具都有独立 system prompt，避免一个超长 prompt。三档结构：

1. **Router Prompt**：极短（< 200 token），只输出 `{"intent": "...", "tools": [...], "confidence": 0.0~1.0}`。
2. **Tool Prompt**：每个 tool 自带 JSON Schema 提示（must_not 槽位强约束）。
3. **Answer Prompt**：约束「只用 context 里出现的 SPU，编号引用 [#1] [#2]」+「禁止编造价格 / 优惠券 / 库存」+「如果信息不足，主动反问」。

→ 主动反问能力 = 进阶场景里的"拍照优先还是性价比？"分数点，必须在 Answer Prompt 里显式 few-shot。

## 5. 后端服务化（评分项 3.3）

### 5.1 流式 API

**保留并复用 retrieval 现有 SSE**。新增 `POST /agent/turn`（SSE）事件类型：

```
event: turn.started      data: {turnId, conversationId}
event: intent.detected   data: {intent, slots}
event: tool.calling      data: {tool, args}
event: tool.result       data: {tool, items: [SpuCard]}  ← 商品卡片
event: answer.delta      data: {text}                    ← 增量 token
event: citation          data: {refId, spuId}            ← 强引用
event: turn.completed    data: {turnId, latencyMs, tokenUsage}
event: error             data: {code, msg, recoverable}
```

> 让客户端能在 LLM 还在生成 token 的时候，就把商品卡片先渲染出来（`tool.result` 在 `answer.delta` 之前到）—— **首屏体验** >
> 字节豆包级别。这是 demo 视频里能直接看出的差异化。

### 5.2 可扩展网关

把所有外部模型/向量库调用收口到一个 `AiGatewayClient`（已有 Spring AI 抽象基础上薄封装），统一处理：

- 限流（令牌桶 + 队列）：尊重 Doubao TPM 80 万 / RPM 700。
- 熔断（resilience4j，可不引入直接用 Reactor `onErrorResume`）。
- 降级路径（chat 不可用时回退到关键词检索 + 模板回答）。
- Embedding 命中现有 `RagEmbeddingCacheRepository`。

### 5.3 服务稳定性

- **超时分级**：路由 600ms / 检索 1.5s / 生成 8s，每一段都打 `RagRetrievalMetrics` 现成的 metric。
- **降级**：检索失败 → 关键词兜底 + "我没找到精确匹配，您是不是想说 …"。
- **幂等**：`turnId` 作为幂等键，存 `agent_turn`（参考 `RagAskRunRecord` 的模式）。
- **恢复任务**：复用 `RagAskRecoveryTask` 的套路，对 stuck turn 自动 fail 并清理。

|--------|------------|-----------------------------------------------------------------------------------------------------------------------------|--------------------------------------------------------------|--------------------------|
| 基础     | 单轮模糊推荐     | `IntentClassifier → search_products → answer`                                                                               | retrieval 全栈                                                 | catalog 数据               |
| 基础     | 条件筛选       | SlotExtractor 抽出 `price<200 + tag=蓝牙` → `RagMilvusNativeExpressionBuilder`                                                  | 现成的过滤构造器                                                     | Slot schema              |
| 进阶     | 多轮追问/细化    | `ConversationMemory` 把上一轮 result 子集传入 SlotExtractor + 主动反问 prompt                                                           | `RagConversationService`                                     | refine tool              |
| 进阶     | 对比决策       | `compare_products` 并发检索 N + 属性表格 + 推荐理由                                                                                     | HybridRetriever                                              | compare tool + 表格 prompt |
| **高级** | **反选/排除**  | SlotExtractor 输出 `must_not.brand=["日系品牌"]`，**Milvus expr + 后排过滤双保险**，UI 显示「已排除」                                             | `RagSearchFilter`/`RagMilvusNativeExpressionBuilder` 现有的过滤通道 | 否定语义 prompt + 后排过滤器      |
| 高级     | 场景化组合      | `scenario_outfit` tool：LLM 把"三亚度假"拆 3 类目 → 并发 3 次检索 → 重排融合 → 卡片组                                                            | 并发检索能力（Reactor）                                              | fan-out planner          |
| **高级** | **购物车与下单** | `CartStateMachine` (复用 Spring Statemachine)：`IDLE→ITEM_PROPOSED→CONFIRMED→PLACED`；对话式 CRUD：add/remove/edit/checkout，全部回滚可审计 | `IndexWorkflow` 同款 SM 套路                                     | cart/order 模块 + 4 个 tool |
| **高级** | **拍照找货**   | `image_search` tool：用户图 → Doubao-embedding-vision → Milvus ANN（同 collection）→ 相似度 ≥ 阈值 → SPU 卡片                             | 现有 Milvus retriever                                          | 多模态上传 + ImageQueryTool   |

### 6.1 反选/排除 — 一等奖的"切胃菜"

否定语义易错的地方在于：

1. **抽取**：用 function_call schema 强制 LLM 把 `must_not.tags / must_not.brands / must_not.ingredients`
   拆出来，而不是塞进一个长字符串里。
2. **过滤**：在 Milvus 表达式生成器里加 `tags not in (...)`，但 LLM 写的属性词不一定能精确匹配 metadata，所以**后排再用 LLM
   做一次"含 X 否"判定**（仅对 top-50 召回做，成本可控）。
3. **可解释**：UI 卡片头显示 `已为您排除：日系品牌 / 含酒精`，让评委看到效果。

### 6.2 购物车 — Statemachine 是字节工程化的体现

```
状态：IDLE → ITEM_PROPOSED → IN_CART → CHECKING_OUT → PLACED / CANCELLED
事件：PROPOSE_ITEM / CONFIRM_ADD / REMOVE / UPDATE_QTY / CHECKOUT / CANCEL
守卫：库存校验 / 价格变动二次确认 / 收货地址有效性
```

复用 `IndexWorkflowStateMachineFactory` 的模式去搭，连 `RagIndexJobTransitionRecord` 这种 transition audit 都能照搬，*
*直接对应"对话式 CRUD 操作 + 多轮状态管理 + 业务闭环"** 这一行评分。

### 6.3 拍照找货 — 多模态的"杀手锏"

```
1. iOS 端 上传图片 + 文字（"我想要同款外套"）→ multipart/form-data
2. 后端：
   a. ImageQueryTool 接收 → Doubao-embedding-vision(image) → vector
   b. Milvus search where chunk_type ∈ {IMAGE, TITLE} order by L2
   c. text query 也 embed 一遍，做 RRF 融合（再合一次重排器）
3. 输出：相似商品 + 用 LLM 生成"这一款也是工装风格 + 同色系"的视觉解释
```

> **Demo 杀手锏**：演示从手机相册选一张街拍 → 3 秒出"同款工装外套" + 类似风格的 2 款 → 加购 → 下单全闭环。这一段视频能直接打
> 95+。

## 11. 现场答辩的"杀手锏"卖点

1. "我们不是单纯做 RAG，是基于 Modulith 把它工程化成 Agent 平台 — 这就是为什么我们敢同时上反选、购物车、拍照三个加分点。"
2. "**文本和图像在同一个 Milvus collection**，一次检索就能跨模态混排，不需要切换索引。"
3. "**否定语义我们做了双保险**：Milvus 表达式 + LLM 后排，UI 还会告诉用户'已为您排除 X'。"
4. "**购物车不是简单 CRUD**，是用 Spring Statemachine 做的状态机，每一次状态迁移都有 audit，支持回滚。"
5. "**全链路 trace**：从 intent → tool → embedding → milvus → answer 每一段都有 span，评委可以现场打开 Kibana 看。"
