# RAG 服务内存优化方案

> 针对 Spring Boot 4 + Spring AI + PostgreSQL + Milvus 可选链路的资源优化技术文档

---

## 背景

**项目现状**：
- Java 25 + Spring Boot 4.0.4
- Spring Modulith 分层，核心模块为 `document / indexing / retrieval / infrastructure`
- 文档内容存储在 PostgreSQL，切片存储在 `rag_chunks`
- Milvus 为可选向量库，默认 `rag.milvus.enabled=false`
- RocketMQ 为可选异步索引链路，默认 `rag.rocketmq.enabled=false`
- 当前代码已经手动创建虚拟线程执行器，但 Spring Boot 全局虚拟线程开关尚需显式开启

**主要挑战**：
- 单文档索引链路会同时持有原文、Markdown blocks、chunk drafts、saved chunks、embedding map 和 Milvus upsert 字段列表
- 虚拟线程执行器为无界提交，如果同时索引多个大文档，会把峰值内存叠加
- embedding 向量会经历 `JSON String -> float[] -> List<Float>`，存在装箱和重复复制
- PostgreSQL chunk 插入当前会为整篇文档拼接一个大 `INSERT ... VALUES ... RETURNING`
- 在线问答会拼接完整上下文 prompt，缺少 token 和字符预算保护

**目标**：
- 避免大文档索引触发 OOM
- 将索引链路峰值内存从“按整篇文档放大”改为“按批次放大”
- 保持问答链路可用，同时控制检索扇出和上下文体积
- 让生产配置具备清晰的内存预算、并发上限和降级策略

## 内存分配规划

### 方案 A：2C2G 单机保守部署

适合开发环境、小流量演示环境或低成本部署。该模式不建议在本机运行 Milvus。

| 组件 | 内存限制 | 说明 |
|------|---------|------|
| 操作系统 | 300MB | 系统保留 |
| PostgreSQL | 450MB | 文档、切片、Outbox、审计表 |
| Java 应用 Heap | 768MB | RAG 服务主堆 |
| Java Native/Metaspace/线程栈 | 250MB | JVM 非堆与网络缓冲 |
| RocketMQ/Milvus | 0MB | 单机低配下禁用或使用外部托管 |
| 缓冲区 | 230MB | 应对 GC、上传、临时对象峰值 |
| **总计** | **1998MB** | 可控 |

### 方案 B：4C8G 标准部署

适合预生产或小规模生产。Milvus 和 RocketMQ 仍建议独立部署。

| 组件 | 内存限制 | 说明 |
|------|---------|------|
| 操作系统 | 600MB | 系统保留 |
| PostgreSQL | 1.5GB | 可与应用同机，但更推荐独立 |
| Java 应用 Heap | 2GB | 可支撑中等文档和检索并发 |
| Java Native/Metaspace/线程栈 | 600MB | JVM 非堆、Netty、HTTP 客户端 |
| 缓冲区 | 3.3GB | 文件峰值、GC、系统缓存 |
| **总计** | **8GB** | 推荐起步规格 |

### JVM 建议

```bash
JAVA_TOOL_OPTIONS="
  -XX:MaxRAMPercentage=55
  -XX:InitialRAMPercentage=25
  -XX:+UseG1GC
  -XX:MaxGCPauseMillis=200
  -XX:+HeapDumpOnOutOfMemoryError
  -XX:HeapDumpPath=/var/log/rag/heapdump.hprof
  -XX:NativeMemoryTracking=summary
"
```

低配 2GB 环境建议改为固定 heap：

```bash
JAVA_TOOL_OPTIONS="
  -Xms256m
  -Xmx768m
  -XX:+UseG1GC
  -XX:+HeapDumpOnOutOfMemoryError
  -XX:HeapDumpPath=/var/log/rag/heapdump.hprof
"
```

## 优化方案

### 1. Spring Boot 4 虚拟线程开关

#### 问题分析

当前代码中已经存在手动创建的虚拟线程执行器：

```java
Executors.newVirtualThreadPerTaskExecutor()
```

这只能保证显式使用该 executor 的 RAG 任务运行在虚拟线程上。Spring Boot 管理的 Web 请求、`applicationTaskExecutor`、`@Async`、部分调度执行链路不会自动切换。

#### 解决方案：开启全局虚拟线程

```properties
spring.threads.virtual.enabled=true
```

建议放入 `application-prod.properties`，开发环境也同步打开，避免环境行为不一致。

#### 配套策略

虚拟线程解决的是阻塞等待成本，不解决业务对象占用内存。因此必须同时保留业务背压：

| 场景 | 建议上限 |
|------|---------|
| 同时索引文档数 | 1 到 2 |
| 同时 embedding 批次 | 2 到 4 |
| 同时 LLM 生成 | 8 到 20 |
| 同时大文件上传 | 2 到 5 |

#### 推荐配置

```properties
spring.threads.virtual.enabled=true

rag.indexing.max-inflight-documents=${RAG_INDEXING_MAX_INFLIGHT_DOCUMENTS:1}
rag.indexing.embedding-max-concurrency=${RAG_INDEXING_EMBEDDING_MAX_CONCURRENCY:2}
rag.retrieval.answer-generation-max-concurrency=${RAG_RETRIEVAL_ANSWER_GENERATION_MAX_CONCURRENCY:8}
```

### 2. 索引链路分批化

#### 问题分析

当前单次索引主要内存峰值来自以下对象叠加：

| 阶段 | 主要对象 | 风险 |
|------|----------|------|
| 文档读取 | `document.content()` | 原文完整驻留 |
| Markdown 解析 | normalized、body、lines、blocks | 字符串和数组复制 |
| 切片 | `List<RagTextChunk>` | 每个 chunk 持有文本和元数据 |
| 草稿 | `List<RagChunkDraft>` | 再复制一份 chunk 文本和 metadata |
| 入库回查 | `List<RagChunkRecord>` | `RETURNING` 后再读回 chunk 文本 |
| embedding | `Map<String, float[]>` | 全文档向量同时驻留 |
| Milvus upsert | ids、contents、metadata、`List<List<Float>>` | 装箱对象大量膨胀 |

在 10MB Markdown 文档、`chunk-size=400`、`overlap=80` 的情况下，chunk 数可能达到数万级。此时“全文档一次性处理”会明显放大堆内存。

#### 解决方案：按批处理 chunk

新增配置：

```properties
rag.indexing.chunk-persist-batch-size=${RAG_INDEXING_CHUNK_PERSIST_BATCH_SIZE:100}
rag.indexing.vector-upsert-batch-size=${RAG_INDEXING_VECTOR_UPSERT_BATCH_SIZE:50}
rag.indexing.max-chunks-per-document=${RAG_INDEXING_MAX_CHUNKS_PER_DOCUMENT:5000}
rag.indexing.max-document-chars=${RAG_INDEXING_MAX_DOCUMENT_CHARS:2000000}
```

#### 改造思路

1. `RagTextChunker` 支持迭代式输出，避免一次返回完整 `List<RagTextChunk>`。
2. `RagIndexingService` 以 batch 为单位执行：
   - chunk draft 转换
   - PostgreSQL 批量插入
   - embedding 获取
   - Milvus upsert
3. 每批处理完成后释放局部引用，避免全篇 chunk 同时驻留。
4. 对单文档最大字符数和最大 chunk 数做硬限制，超限直接拒绝或进入人工处理。

#### 伪代码

```java
for (List<RagTextChunk> chunkBatch : chunker.chunkBatches(document.content(), batchSize)) {
    List<RagChunkDraft> drafts = toDrafts(document, chunkBatch, generation);
    List<RagChunkRecord> saved = chunkRepository.saveAll(documentId, drafts);
    milvusVectorIndexer.addBatch(document, saved);
}
```

#### 预期收益

| 优化项 | 优化前 | 优化后 |
|--------|--------|--------|
| chunk 对象 | 全文档驻留 | 单批驻留 |
| embedding map | 全文档驻留 | 单批驻留 |
| Milvus 字段列表 | 全文档驻留 | 单批驻留 |
| OOM 风险 | 随文档大小线性放大 | 受 batch size 控制 |

### 3. Milvus 与 embedding 内存优化

#### 问题分析

当前 Milvus 写入会将 `float[]` 转成 `List<Float>`。`Float` 是对象，1024 维向量会产生 1024 个装箱对象。假设一次 upsert 1000 个 chunk，仅 embedding 装箱对象就可能超过 100 万个。

此外 embedding cache 使用 JSON 文本存储，读取时需要：

```text
embedding_json String -> float[] -> List<Float>
```

#### 解决方案：小批 upsert + 缓存格式优化

短期：

```properties
rag.indexing.vector-upsert-batch-size=50
rag.indexing.embedding-batch-size=10
```

中期：

| 项目 | 方案 |
|------|------|
| embedding cache | 从 JSON 文本改为 `bytea` 或压缩二进制 |
| Milvus upsert | 优先使用 SDK primitive vector API，减少 `Float` 装箱 |
| cache 查询 | 按 batch 查询，避免一次 IN 太多 chunk hash |
| cache 写入 | 批量 upsert，避免逐条 JDBC update |

#### 缓存表建议

```sql
ALTER TABLE rag_embedding_cache
    ADD COLUMN IF NOT EXISTS embedding_bytes BYTEA;
```

迁移策略：

1. 新写入同时写 `embedding_json` 和 `embedding_bytes`。
2. 读取优先使用 `embedding_bytes`。
3. 稳定后移除 JSON 读取路径，或保留为兼容字段。

### 4. PostgreSQL 写入与查询优化

#### 问题分析

当前 PostgreSQL chunk 插入会为整篇文档拼接一个大 SQL：

```sql
INSERT INTO rag_chunks (...) VALUES (...), (...), (...) RETURNING ...
```

当 chunk 数很大时，SQL 字符串、参数数组、返回结果都会增加 JVM 内存峰值。

#### 解决方案：固定批量 + 精简返回字段

建议新增：

```properties
rag.jdbc.chunk-insert-batch-size=${RAG_JDBC_CHUNK_INSERT_BATCH_SIZE:100}
rag.jdbc.embedding-cache-upsert-batch-size=${RAG_JDBC_EMBEDDING_CACHE_UPSERT_BATCH_SIZE:100}
```

`RETURNING` 只保留后续必须字段：

```sql
RETURNING id, document_id, index_generation, chunk_index, chunk_hash, vector_id
```

如果 Milvus 写入仍需要 `chunk_text`，优先直接复用当前 batch 的 `RagChunkDraft`，不要从数据库再回读一份完整文本。

#### 查询侧控制

| 查询 | 当前风险 | 优化方案 |
|------|----------|----------|
| `findSearchableByVectorIds` | IN 参数随候选数增长 | 限制 candidateTopK，必要时分批 |
| 邻居窗口扩展 | 每个 seed 单独查询 | 合并为 documentId + range 批量查询 |
| keyword fallback | LIKE 兜底可能大范围扫描 | 保持 topK 上限，增加 search_vector |

### 5. 文档上传与 Markdown 解析优化

#### 问题分析

生产配置目前默认：

```properties
spring.servlet.multipart.max-file-size=10MB
spring.servlet.multipart.max-request-size=20MB
```

10MB Markdown 在 JVM 中不会只占 10MB。解析过程中会产生 normalized 字符串、body 字符串、lines 数组、blocks、tables、links、metadata 等临时对象。

#### 解决方案：输入限制 + 结构限制

建议新增：

```properties
rag.document.max-content-chars=${RAG_DOCUMENT_MAX_CONTENT_CHARS:2000000}
rag.document.max-line-count=${RAG_DOCUMENT_MAX_LINE_COUNT:50000}
rag.document.max-table-row-count=${RAG_DOCUMENT_MAX_TABLE_ROW_COUNT:5000}
rag.document.max-link-count=${RAG_DOCUMENT_MAX_LINK_COUNT:2000}
```

上传侧保守配置：

```properties
spring.servlet.multipart.max-file-size=${SPRING_SERVLET_MULTIPART_MAX_FILE_SIZE:5MB}
spring.servlet.multipart.max-request-size=${SPRING_SERVLET_MULTIPART_MAX_REQUEST_SIZE:6MB}
```

#### 降级策略

| 超限类型 | 行为 |
|----------|------|
| 内容为空 | 拒绝 |
| 字符数超限 | 返回 413 或业务错误码 |
| 表格行数超限 | 保留原始表格块，跳过 row-description 展开 |
| 链接数超限 | 停止提取 links metadata |
| chunk 数超限 | 标记 FAILED，提示需要拆分文档 |

### 6. 检索与回答生成内存优化

#### 问题分析

在线问答链路包含：

1. query transform
2. query expansion
3. multi-query retrieval
4. hybrid retrieval
5. neighbor window expansion
6. prompt 拼接
7. ChatClient 调用

当前默认 `query-expansion.number-of-queries=3`，每路 query 还会进行 semantic + keyword 检索。候选上限虽然已有配置，但最终 prompt 仍可能拼接较多 chunk 内容。

#### 解决方案：上下文预算

新增配置：

```properties
rag.retrieval.max-context-chars=${RAG_RETRIEVAL_MAX_CONTEXT_CHARS:12000}
rag.retrieval.max-context-tokens=${RAG_RETRIEVAL_MAX_CONTEXT_TOKENS:3000}
rag.retrieval.max-history-messages=${RAG_RETRIEVAL_MAX_HISTORY_MESSAGES:12}
rag.retrieval.max-question-chars=${RAG_RETRIEVAL_MAX_QUESTION_CHARS:2000}
```

生成 prompt 前执行预算裁剪：

```java
List<RagRetrievedChunk> budgetedContexts = contextBudgeter.trim(contexts, maxTokens, maxChars);
```

生产保守参数：

```properties
rag.query-expansion.number-of-queries=2
rag.retrieval.keyword-candidate-top-k-max=50
rag.retrieval.semantic-candidate-top-k-max=30
rag.retrieval.multi-query-per-query-top-k-max=8
rag.retrieval.neighbor-window-before=1
rag.retrieval.neighbor-window-after=1
```

极低内存场景：

```properties
rag.query-expansion.enabled=false
rag.query-transformation.enabled=false
rag.retrieval.neighbor-window-before=0
rag.retrieval.neighbor-window-after=0
```

### 7. 向量检索部署优化

#### 问题分析

Milvus 对内存和后台线程有基础成本，不适合 2GB 单机与 Java 应用、PostgreSQL 混部。

#### 解决方案：分层选型

| 场景 | 推荐方案 |
|------|----------|
| 2C2G 单机 | 禁用本机 Milvus，仅使用 keyword retrieval |
| 4C8G 小规模 | Milvus 外部托管或独立机器 |
| 成本敏感中等规模 | 评估 PostgreSQL + pgvector |
| 高召回质量要求 | Milvus 独立部署 |

#### 当前配置建议

低配环境：

```properties
rag.milvus.enabled=false
rag.rocketmq.enabled=false
```

标准环境：

```properties
rag.milvus.enabled=true
rag.milvus.uri=${RAG_MILVUS_URI}
rag.rocketmq.enabled=true
```

### 8. 日志与审计内存优化

#### 问题分析

RAG 链路日志如果打印完整 question、context、payload、metadata，会引入额外字符串驻留和日志 IO 压力。

当前项目已有 `RagLogHelper.previewQuestion` 和短 SHA 打印，应继续保持“预览而非全文”的策略。

#### 解决方案

```properties
logging.level.com.involutionhell.backend.rag=INFO
logging.level.com.involutionhell.backend.rag.indexing=INFO
logging.level.com.involutionhell.backend.rag.retrieval=INFO

rag.rocketmq.parse-failure-payload-preview-length=1000
```

建议新增：

```properties
rag.logging.max-question-preview-length=120
rag.logging.max-error-message-length=400
rag.logging.max-metadata-preview-length=1000
```

## 故障恢复

### OOM 预防

1. 启用 `HeapDumpOnOutOfMemoryError`。
2. 索引任务执行前检查文档大小和 chunk 预估数。
3. 获取索引并发令牌失败时返回繁忙状态或进入排队。
4. embedding 和 Milvus 写入按 batch 执行。
5. prompt 构建前强制裁剪上下文。

### 优雅降级

| 故障 | 降级策略 |
|------|----------|
| Milvus 不可用 | 降级 keyword retrieval |
| ChatModel 不可用 | 返回检索摘要 |
| embedding 限流 | 索引任务重试或进入 FAILED |
| PostgreSQL 慢查询 | 降低 candidateTopK，关闭 query expansion |
| 内存接近阈值 | 暂停新索引任务，只保留查询 |

## 总结

### 优化效果

| 优化项 | 预期收益 |
|--------|----------|
| 开启 Spring Boot 4 虚拟线程 | 降低阻塞请求线程成本 |
| 索引并发背压 | 避免多个大文档同时放大内存 |
| chunk 分批处理 | 峰值内存从全文档级降到 batch 级 |
| Milvus batch upsert | 减少 `Float` 装箱对象峰值 |
| PostgreSQL batch insert | 避免巨型 SQL 和大结果回读 |
| prompt budget | 避免长上下文拖垮堆内存和模型调用 |

### 关键技术

- Spring Boot 4 virtual threads
- G1GC heap budget
- Semaphore 背压
- Batch indexing pipeline
- Context token budget
- Actuator + JFR + NMT 观测

### 适用场景

- 低配单机 RAG 服务
- 文档量持续增长但希望控制成本的知识库服务
- Milvus 可选、PostgreSQL 常驻的中小型 RAG 应用
- 需要避免索引峰值 OOM 的 Spring Boot 服务
